# تحليل مشروع لعبة "الأسماء السرية" (Codenames P2P)

> [!NOTE]
> هذا الملف تم تحديثه بعد مراجعة ثانية شاملة لجميع ملفات المشروع. التحليل الأول كان ناقصاً في 5 نقاط رئيسية مذكورة في نهاية هذا المستند.

---

## 1. صفحات المشروع (4 صفحات)

يتكون المشروع من **أربع** صفحات، وليس ثلاث كما ذُكر سابقاً:

### أ. شاشة البداية (Start Screen)
*   **الوظيفة:** نقطة الدخول الرئيسية للتطبيق.
*   **المكونات:**
    *   حقل إدخال اسم اللاعب.
    *   خيار "إنشاء غرفة" ليصبح المستخدم هو **المضيف (Host)**.
    *   حقل إدخال عنوان IP وخيار "انضمام" ليصبح المستخدم **منضماً (Joiner)**.
    *   زر للوصول إلى "الاختبار الذاتي".
*   **الحالة التقنية:** تستخدم `Hive` لحفظ آخر IP وآخر اسم مستخدم لتسهيل الدخول المرة القادمة.

### ب. شاشة الانتظار (Lobby Screen)
*   **الوظيفة:** تنظيم اللاعبين قبل بدء اللعبة.
*   **المكونات:**
    *   عرض رقم الغرفة (IP) الخاص بالمضيف.
    *   قائمتان جنباً إلى جنب للفريق الأحمر والأزرق.
    *   بطاقة إعدادات اللاعب المحلي: قائمتان منسدلتان لاختيار الفريق والدور.
    *   زر "بدء اللعبة" (يظهر للمضيف فقط) — ويتطلب: (1) 4 لاعبين على الأقل، (2) رئيس شبكة واحد في كل فريق.
    *   نص "بانتظار المضيف..." يظهر بدلاً من الزر لغير المضيفين.

> [!IMPORTANT]
> **قاعدة الـ Spymaster الفريدة (مخفية في الكود):** عند الانضمام، إذا حاول لاعب اختيار دور رئيس الشبكة وكان هناك رئيس بالفعل في نفس الفريق، **يتم إسناده تلقائياً إلى دور عميل ميداني** دون إشعار. هذه قاعدة غير ظاهرة في الواجهة.

### ج. شاشة لوحة اللعب (Game Board Screen)
*   **الوظيفة:** الساحة الرئيسية التي تتم فيها اللعبة.
*   **المكونات:**
    *   **شريط النتائج:** يظهر العداد الأحمر (9 كروت) والأزرق (8 كروت) والدور الحالي بلونه.
    *   **شبكة الكلمات:** 25 كارت (5×5) — تختلف ألوانها بناءً على دور اللاعب.
    *   **منطقة التحكم الديناميكية:** تتغير كلياً بناءً على الدور والحالة (4 حالات مختلفة مشروحة في القسم التالي).
    *   **شريط النهاية:** يظهر عند انتهاء اللعبة مع زر "إعادة اللعب" للمضيف فقط.

### د. شاشة الاختبار الذاتي (Test Runner Screen)
*   **الوظيفة:** أداة تشخيصية للمطورين فقط، لا يستخدمها اللاعبون العاديون.
*   **الوصول:** زر "اختبار ذاتي 🧪" (بنفسجي اللون) في أسفل شاشة البداية.
*   **المكونات:**
    *   شريط ملخص (الكلي/النجاح/الفشل).
    *   قائمة بـ16 اختباراً موزعة على 6 مجموعات: Engine, Network, Validators, Storage, State, Edge Cases.
    *   زر "Run All Tests" لتشغيل جميع الاختبارات بشكل متسلسل.
*   **الملاحظة:** الشاشة مكتوبة بالإنجليزية، بينما باقي الواجهات بالعربية — هذا تناقض بصري واضح.

## 2. أنواع المستخدمين ورحلاتهم (User Journeys)

تتقاطع 4 أدوار: دوران شبكيان (Host/Joiner) × دوران لعبيان (Spymaster/Operative) → ينتج 4 تجارب مستخدم مختلفة.

### 👑 المضيف (Host)
| المرحلة | التجربة |
|---|---|
| البداية | يدخل الاسم فقط → زر "إنشاء غرفة" → السيرفر يبدأ فوراً على **Port 4567** |
| الانتقال | يُنقل تلقائياً لشاشة الانتظار (يُسند له افتراضياً: الفريق الأحمر + رئيس شبكة) |
| الانتظار | يرى رقم غرفته (IP)، يوزع الفرق، يُعدّل دوره |
| البدء | زر "بدء اللعبة" يُرسل إشارة START_GAME لجميع اللاعبين في آنٍ واحد |
| اللعب | يلعب كأي لاعب + يملك صلاحية الـ Rematch |

### 🤝 اللاعب المنضم (Joiner)
| المرحلة | التجربة |
|---|---|
| البداية | يدخل اسمه + IP المضيف → زر "انضمام" |
| التحقق | التطبيق يتحقق من صحة الـ IP قبل الاتصال (تحقق بالـ regex) |
| الانضمام | يُنقل لشاشة الانتظار ويتلقى فوراً قائمة اللاعبين الحاليين |
| الانتظار | يختار فريقه ودوره؛ يرى "بانتظار المضيف..." بدلاً من زر البدء |
| البدء | يُنقل تلقائياً للعب عند إرسال المضيف إشارة البدء |

### 🔍 رئيس الشبكة (Spymaster)

**ما يراه الـ Spymaster في لوحة اللعب:**
- كروت فريقه: **خلفية حمراء/زرقاء شفافة (20% opacity)** مع النص الملون
- كروت المحايد: **بنية شفافة**
- كروت القاتل (Assassin): **سوداء شفافة**
- الكروت المكشوفة: **ألوانها الحقيقية مع خط شطب**

**منطقة التحكم — 3 حالات مختلفة:**
1. **ليس دور فريقه:** → نص رمادي "بانتظار دور فريقك..."
2. **دوره + لم يعطِ تلميحاً بعد:** → حقل نصي للتلميح + قائمة منسدلة للعدد (0→9 + ∞) + زر إرسال
3. **دوره + أعطى التلميح:** → نص "لقد أعطيت تلميحاً! بانتظار العملاء..."

### 🔫 العميل الميداني (Operative)

**ما يراه الـ Operative في لوحة اللعب:**
- جميع الكروت غير المكشوفة: **بيج (amber.shade100)** — لا ألوان
- الكروت القابلة للضغط: **إطار أخضر لامع** حول الكارت
- الكروت المكشوفة: **ألوانها الحقيقية مع خط شطب**

**منطقة التحكم — 3 حالات مختلفة:**
1. **لم يصدر تلميح بعد:** → نص "الرئيس يفكر بالتلميح..."
2. **وصل التلميح + دور فريقه:** → بطاقة صفراء تعرض التلميح والعدد والمتبقي + **زر "إنهاء الدور"** (بعد أول تخمين)
3. **ليس دور فريقه:** → يرى التلميح لكن بدون عناصر تفاعلية

## 3. طرق الانتقال والتدفق (Navigation Flow)

```
StartScreen
   ├── [إنشاء غرفة] ──────────────────► LobbyScreen (isHost=true)
   │      (يبدأ السيرفر على Port 4567)         │
   │                                            ├─ [المضيف يضغط بدء اللعبة]
   └── [انضمام]  ──────────────────────► LobbyScreen (isHost=false)        │
          (يتصل بسيرفر المضيف)                 │                             ▼
                                                └─ [تلقائي عبر P2P] ──► GameBoardScreen
                                                                              │
StartScreen ◄──── [زر رجوع + disconnect()] ──────────────────────────────────┘

StartScreen ──── [زر الاختبار الذاتي] ──► TestRunnerScreen (مكدس فوق StartScreen)
```

### تفاصيل الإشارات (Socket Messages)

| الإشارة | من → إلى | الوظيفة |
|---|---|---|
| `REQ_JOIN` | Client → Host | طلب الانضمام مع الاسم والـ ID |
| `LOBBY_UPDATE` | Host → All | تحديث قائمة اللاعبين |
| `PLAYER_UPDATE` | Client → Host | تغيير الفريق أو الدور |
| `START_GAME` | Host → All | بدء اللعبة وتحويل الشاشة |
| `STATE_UPDATE` | Host → Client | تحديث حالة اللعبة (مُعدَّلة حسب الدور) |
| `CARD_FLIP` | Client → Host | طلب كشف كارت |
| `CLUE_GIVEN` | Client/Host → Host | إرسال تلميح |
| `PASS_TURN` | Client/Host → Host | تمرير الدور |

> [!IMPORTANT]
> **أمان توزيع البيانات:** عند إرسال `STATE_UPDATE` للعملاء الميدانيين، يقوم الـ Host بإخفاء ألوان الكروت غير المكشوفة (يُحوّلها كلها إلى `neutral`). أي أن **الـ Operative لا يحصل أبداً على الألوان الحقيقية** من الشبكة. هذا منطق أمان فعلي في الكود.

### قواعد اللعبة المطبقة في الكود

- **التوزيع:** 9 كروت حمراء، 8 زرقاء، 7 محايدة، 1 قاتل (Assassin) = 25 كارت إجمالاً.
- **الكروت التلميح:** العدد المسموح = العدد المختار + 1 (مثلاً: 3 → 4 تخمينات).
- **99 (∞):** يعني تخمينات غير محدودة.
- **اكتشاف الفوز:** أحمر → 9 كروت مكشوفة؛ أزرق → 8 كروت. (الأحمر يبدأ أولاً لذا له كارت إضافي).
- **اكتشاف القاتل:** يخسر الفريق المكتشِف فوراً، والفريق الآخر يفوز.
- **إنهاء الدور إجبارياً:** عند اختيار كارت محايد أو كارت العدو.

## 4. الأخطاء والثغرات في التحليل الأول

> [!WARNING]
> **5 نقاط كانت غائبة أو مغلوطة في التحليل الأول:**

1. **الشاشة الرابعة مفقودة:** لم يُذكر [TestRunnerScreen](file:///c:/Users/gaa77/Desktop/pro2/p2p/lib/features/testing/presentation/test_runner_screen.dart#12-18) بالتفصيل مع أنه صفحة كاملة مع قائمة اختبارات وواجهة مستقلة.
2. **إعادة اللعبة (Rematch) غير مكتملة:** في التحليل الأول ذُكر أن المضيف "يرسل إشعاراً" للجميع — لكن الواقع أن زر إعادة اللعب لديه `// TODO` في الكود؛ أي أن **إعادة اللعب تُصفّر اللوحة محلياً للمضيف فقط** ولا تُزامن تلقائياً للمنضمين.
3. **منطق الأمان مفقود:** لم يُذكَر أن الـ Host يُعدّل البيانات قبل إرسالها للـ Operative لإخفاء الألوان الحقيقية ([_sendGameStateToPlayer](file:///c:/Users/gaa77/Desktop/pro2/p2p/lib/core/network/socket_host.dart#150-168) + redaction).
4. **قاعدة رئيس الشبكة الوحيد:** لم يُذكَر أن السيرفر يمنع وجود أكثر من رئيس شبكة واحد في الفريق ويُخضع الثاني لدور عميل تلقائياً.
5. **لم تُذكر قواعد اللعبة الفعلية:** توزيع الكروت (9/8/7/1)، منطق النقاط، وطريقة حساب التخمينات.
